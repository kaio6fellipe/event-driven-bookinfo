# API Spec Generation & Backstage Event Catalog

**Status:** Design approved · Implementation pending
**Date:** 2026-04-25
**Scope:** All services in `services/*/` (ratings, details, reviews, ingestion, notification, dlqueue, productpage)

## Problem

The repo has no machine-readable description of any service's HTTP API or the
events each service publishes to Kafka. Three downstream needs cannot be met
without one:

1. **Documentation** — readable references for the seven services for humans
   browsing the system.
2. **Helm values generation** — `cqrs.endpoints.<key>.{method, endpoint}` and
   `events.exposed.<key>.{contentType, eventTypes}` are currently hand-edited
   per service in `deploy/<svc>/values-local.yaml` and drift from the Go code
   that they describe. They should be derived from the code.
3. **Backstage event catalog (external IDP)** — a separate private repo
   running Backstage hosts a scaffolding template that lets other teams browse
   exposed events and click "configure to my service" to wire a consumer. That
   template needs a discoverable, well-typed catalog of every event this
   monorepo emits, with the JSONSchema of each payload.

A fourth concern is contract integrity: any breaking change to a published
event or REST endpoint silently rolling out to consumers is the highest-cost
failure mode for an IDP-driven org.

## Goals

- Single source of truth in Go code for both REST endpoints and exposed events.
- Per-service `openapi.yaml` (OpenAPI 3.1) and `asyncapi.yaml` (AsyncAPI 3.0)
  artifacts, committed to the repo, regenerated by a single command.
- Per-service `catalog-info.yaml` for Backstage GitHub discovery (one
  `kind: Component`, one `kind: API` per spec).
- Per-service `deploy/<svc>/values-generated.yaml` containing the structural
  (spec-derived) keys of `cqrs.endpoints` and `events.exposed`, deep-merged at
  install time with the env-specific `values-local.yaml`.
- CI gates: drift detection, lint, breaking-change detection.

## Non-goals

- Client SDK generation (oapi-codegen / asyncapi-codegen-go). Architecturally
  trivial once the specs exist — deferred to a follow-up plan.
- Tying spec `info.version` to per-service git tags. Possible but couples
  spec lifecycle to release tooling — deferred.
- Pact / contract testing harness driven by the specs. Deferred.
- AsyncAPI spec-first or OpenAPI handler regeneration. Code is the source of
  truth on both sides; specs are derived.

## Architecture

```
services/<svc>/internal/adapter/inbound/http/endpoints.go     ┐
  Endpoints []api.Endpoint                                   │
                                                             │   tools/specgen
services/<svc>/internal/adapter/outbound/kafka/exposed.go    ├──────────────►  services/<svc>/api/openapi.yaml
  Exposed []events.Descriptor                                │   (Go AST +      services/<svc>/api/asyncapi.yaml
                                                             │    invopop/    services/<svc>/api/catalog-info.yaml
                                                             │    jsonschema) deploy/<svc>/values-generated.yaml
                                                             ┘
```

Two declarative slices in each service's source tree describe everything the
generator needs. Handlers register routes by looping over the same slice that
the generator reads, so the slice is also load-bearing for runtime behavior —
spec drift would break the service before it could be deployed, which is the
strongest possible drift guarantee.

## Source-of-truth conventions

The two declarative slices live next to the types they reference, so neither
package needs to cross hex-arch boundaries to import the others' DTOs.

### `services/<svc>/internal/adapter/inbound/http/endpoints.go`

```go
package http

import "github.com/kaio6fellipe/event-driven-bookinfo/pkg/api"

const APIVersion = "1.0.0" // surfaced as info.version in openapi.yaml

var Endpoints = []api.Endpoint{
    {
        Method:   "GET",
        Path:     "/v1/ratings/{id}",
        Summary:  "Get all ratings for a product",
        Response: ProductRatingsResponse{},
        Errors:   []api.ErrorResponse{{Status: 500, Type: ErrorResponse{}}},
    },
    {
        Method:    "POST",
        Path:      "/v1/ratings",
        Summary:   "Submit a new rating",
        EventName: "rating-submitted", // ties to cqrs.endpoints.<key>
        Request:   SubmitRatingRequest{},
        Response:  RatingResponse{},
        Errors:    []api.ErrorResponse{{Status: 400, Type: ErrorResponse{}}},
    },
}
```

`pkg/api` defines the `Endpoint` and `ErrorResponse` types and a small helper
that wires the slice into a `*http.ServeMux`. Existing handlers'
`RegisterRoutes` becomes a one-liner loop over `Endpoints`. The `APIVersion`
constant is the source of truth for `info.version` in the generated
`openapi.yaml`; bumping it is what triggers semver tracking and (combined with
`oasdiff`) gates the breaking-change CI job.

The optional `EventName` field is the join key between an OpenAPI POST handler
and the corresponding `cqrs.endpoints.<EventName>` block in the generated Helm
values. POSTs without an `EventName` are documented in OpenAPI but produce no
`cqrs.endpoints` entry.

**productpage** is a special case: handlers return `text/html` (HTMX), not
JSON. The generator records an OpenAPI operation with `responses.200.content`
keyed by `text/html` and no schema, which is valid OpenAPI 3.1.

### `services/<svc>/internal/adapter/outbound/kafka/exposed.go`

```go
package kafka

import "github.com/kaio6fellipe/event-driven-bookinfo/pkg/events"

var Exposed = []events.Descriptor{
    {
        Name:        "book-added",
        ExposureKey: "events",                     // groups into events.exposed.events
        CEType:      "com.bookinfo.details.book-added",
        CESource:    "details",
        Version:     "1.0",
        ContentType: "application/json",
        Payload:     BookAddedEvent{},             // exported producer-side payload struct
        Description: "Emitted when a new book detail record is created.",
    },
}
```

`pkg/events` defines `Descriptor` and a constructor that the producer uses to
build CloudEvents records (`Producer.Publish(ctx, descriptor, payload)`),
replacing the current hardcoded `ceType`/`ceSource`/`ceVersion` constants in
`services/ingestion/internal/adapter/outbound/kafka/producer.go`. The producer
refactor is part of each service's migration PR. Same drift guarantee as
endpoints: the producer reads the same slice the generator reads.

`ExposureKey` is the grouping key for `events.exposed.<key>` in the chart.
Multiple descriptors with the same `ExposureKey` are emitted under one Helm
block whose `eventTypes` array collects all their `CEType`s. This matches the
existing chart shape where one EventSource publishes multiple CE types from
one Kafka topic. `ExposureKey` defaults to `Name` if omitted.

The unexported `bookEvent` struct in the existing producer becomes an exported
`BookAddedEvent` so the descriptor can reference it. No semantic change.

## Generator: `tools/specgen/`

Single Go binary, repo-internal. Subcommands:

| Subcommand     | Action                                                                                                                |
|----------------|------------------------------------------------------------------------------------------------------------------------|
| `specgen all`  | Walks all services, regenerates all four artifacts per service.                                                       |
| `specgen lint` | Runs spectral rulesets against every `openapi.yaml` and `asyncapi.yaml`, plus Backstage catalog-info schema check.    |
| `specgen diff` | Compares specs against `origin/main` head: oasdiff `breaking` for OpenAPI; advisory diff for AsyncAPI.                |

Implementation:

- `go/packages` + `go/types` to load each service's
  `internal/adapter/inbound/http` and `internal/adapter/outbound/kafka`
  packages and locate the `Endpoints` and `Exposed` slices.
- Reflectively walks the slice contents at AST level (literal composite
  expressions) — does not run the service binary.
- Resolves each referenced struct type and produces a JSONSchema via
  `invopop/jsonschema` (already idiomatic in the Go ecosystem; respects
  `json:` and `validate:` struct tags).
- Composes OpenAPI 3.1 and AsyncAPI 3.0 documents with the schemas as
  reusable components.

## Generated artifacts

### `services/<svc>/api/openapi.yaml`

OpenAPI 3.1, one path per `Endpoints` entry, request/response schemas in
`components.schemas`. `info.version` is the `APIVersion` constant from
`endpoints.go`. Header comment marks the file as generated and points to the
source slice.

### `services/<svc>/api/asyncapi.yaml`

AsyncAPI 3.0, one channel per descriptor `ExposureKey` group. Each channel:
- `address`: the `ExposureKey` (matches the Helm `events.exposed.<key>`)
- `messages.<descriptor.Name>.payload`: JSONSchema of `Payload` struct
- `messages.<descriptor.Name>.contentType`: from descriptor
- `messages.<descriptor.Name>.headers`: CloudEvents binary headers (`ce-type`,
  `ce-source`, `ce-specversion`)

When multiple descriptors share an `ExposureKey`, the channel exposes one
message per descriptor, and consumers discriminate on the `ce-type` header.
This is exactly the existing chart wiring.

Topic and broker are deliberately omitted from the spec (env-specific) and
remain in `values-local.yaml`.

### `services/<svc>/api/catalog-info.yaml`

```yaml
apiVersion: backstage.io/v1alpha1
kind: Component
metadata:
  name: details
  annotations:
    backstage.io/source-location: url:https://github.com/kaio6fellipe/event-driven-bookinfo/tree/main/services/details
spec:
  type: service
  lifecycle: experimental
  owner: bookinfo-team
  providesApis:
    - details-rest
    - details-events
  consumesApis: []
---
apiVersion: backstage.io/v1alpha1
kind: API
metadata:
  name: details-rest
spec:
  type: openapi
  lifecycle: experimental
  owner: bookinfo-team
  definition:
    $text: ./openapi.yaml
---
apiVersion: backstage.io/v1alpha1
kind: API
metadata:
  name: details-events
  annotations:
    bookinfo.io/exposed-event-names: "book-added"
spec:
  type: asyncapi
  lifecycle: experimental
  owner: bookinfo-team
  definition:
    $text: ./asyncapi.yaml
```

The `bookinfo.io/exposed-event-names` annotation is a comma-separated list of
descriptor names — a coarse index for catalog search filters. The Backstage
scaffolding plugin parses the AsyncAPI itself for the data it needs to wire a
consumer: each consumer block requires `eventSourceName` (derived from
`<service>-<ExposureKey>`, matching the chart's EventSource naming
convention), `eventName` (= `ExposureKey`), and an optional `filter.ceType`
(= the descriptor's `CEType`). All three are present in the AsyncAPI doc.

### `deploy/<svc>/values-generated.yaml`

```yaml
# DO NOT EDIT — generated by tools/specgen from
#   services/<svc>/internal/adapter/inbound/http/endpoints.go
#   services/<svc>/internal/adapter/outbound/kafka/exposed.go
# Run `make generate-specs` to refresh.
cqrs:
  endpoints:
    rating-submitted:        # from Endpoints[i].EventName
      method: POST           # from Endpoints[i].Method
      endpoint: /v1/ratings  # from Endpoints[i].Path
events:
  exposed:
    events:                  # from Exposed[*].ExposureKey (grouping key)
      contentType: application/json
      eventTypes:            # union of all descriptors sharing the ExposureKey
        - com.bookinfo.details.book-added
```

The generator owns only these subkeys. Everything else (`port`, `triggers`,
`topic`, `eventBusName`, broker addresses, dlq URLs, replicas, observability
endpoints) stays in `values-local.yaml`.

## Helm wiring

Every install command becomes a dual `-f`:

```bash
helm upgrade --install ratings charts/bookinfo-service \
  -f deploy/ratings/values-generated.yaml \
  -f deploy/ratings/values-local.yaml \
  -n bookinfo
```

Order matters: `values-local.yaml` is loaded last so a hand-edit can still
override generated values in emergencies. Disjoint subkeys mean this never
matters in practice.

`Makefile` and the `make run-k8s` / `make k8s-rebuild` targets are updated to
the dual-`-f` form. Backstage scaffolding writes `events.consumed.<name>` into
`values-local.yaml` (which it owns end-to-end), so its scaffolding output
remains untouched.

## Automation

| Target / Job          | Action                                                                                                                            |
|-----------------------|------------------------------------------------------------------------------------------------------------------------------------|
| `make generate-specs` | Builds `tools/specgen` if stale, runs `specgen all`, writes all artifacts.                                                        |
| `make lint-specs`     | `specgen lint` — spectral OpenAPI 3.1 + AsyncAPI 3.0 rulesets, plus Backstage catalog-info schema.                                |
| `make diff-specs`     | `specgen diff` — oasdiff `breaking` for OpenAPI vs `origin/main`; AsyncAPI advisory diff.                                         |
| CI `specs-drift`      | Runs `make generate-specs && git diff --exit-code`. Fails if devs forgot to regenerate.                                          |
| CI `specs-lint`       | Runs `make lint-specs`.                                                                                                            |
| CI `specs-breaking`   | Runs `make diff-specs`. Fails on breaking OpenAPI change unless PR carries the `breaking-change` label.                           |

CI jobs trigger on PRs touching `services/**`, `tools/specgen/**`, `pkg/api/**`,
`pkg/events/**`, or `deploy/**`.

## Migration

Rollout is per-service, low-risk first. Each migration is a self-contained PR.

1. **Foundation** — build `tools/specgen`, add `pkg/api`, `pkg/events`,
   `make generate-specs`, `make lint-specs`, `make diff-specs`, and the three
   CI jobs. No service changes yet. Specs not committed.
2. **ratings** — smallest service, no exposed events. Refactors handler
   `RegisterRoutes` into an `Endpoints` slice; runs the generator; commits
   `services/ratings/api/{openapi,catalog-info}.yaml` and
   `deploy/ratings/values-generated.yaml`. Validates the OpenAPI half +
   drift gate end-to-end.
3. **ingestion** — producer-only, no HTTP business API. Validates AsyncAPI
   side and Backstage catalog entity layout.
4. **details** — both halves: emits `book-added`, consumes `raw-books-details`,
   has CQRS endpoint. Validates full pipeline including `values-generated.yaml`
   deep-merge with existing `values-local.yaml`. Includes the dual-`-f` switch
   in `Makefile` for k8s deploys; remaining services switch incrementally.
5. **reviews, notification, dlqueue, productpage** — in parallel once the
   pattern is proven.

Rollback granularity is one service's PR.

## Risks & open questions

- **AsyncAPI breaking-change tooling**: `asyncapi diff` for v3 is immature.
  Using it as advisory rather than a hard gate; revisit when the tool
  matures or substitute a custom diff if drift becomes a problem.
- **JSONSchema fidelity**: `invopop/jsonschema` covers common Go idioms but
  has gaps on complex generic types and `time.Time` representation. Address
  per-case via custom `JSONSchemaSchema` methods on affected DTOs.
- **Backstage discovery glob**: depends on the exact GitHub-discovery config
  in the IDP repo. The format above is the documented contract; final glob
  is set in the IDP repo and is not changed by this work.
- **Generated file pollution in PR diffs**: every spec change shows up as
  YAML churn alongside Go changes. Acceptable trade-off — the alternative
  (gitignored specs) hides spec changes from review entirely, which works
  against the contract-as-source-of-truth goal.

## Skipped: productpage

`productpage` is the only service in this monorepo that does not
participate in the generated API catalog. The decision is intentional:

- **It is a BFF (HTML + HTMX), not a service-to-service API surface.**
  The handlers under `services/productpage/internal/handler/` return
  `text/html` for the user-facing pages and HTMX partials. The single
  REST endpoint `GET /v1/products/{id}` is consumed by productpage's own
  frontend (HTMX poll), not by other services in the system.
- **Its layout does not match the discovery contract.** `tools/specgen`
  walks `services/<svc>/internal/adapter/inbound/http/endpoints.go` and
  `services/<svc>/internal/adapter/outbound/kafka/exposed.go`.
  productpage uses a flat `internal/handler/` layout (no hex-arch
  inbound/outbound split) and has no kafka producer, so neither slice
  has a natural home there without restructuring the package.
- **The Backstage event-catalog scaffolding template targets
  cross-team integration use cases.** A team browsing the catalog and
  clicking "configure to my service" wants to wire a *consumer* to one
  of the events exposed elsewhere in the system. productpage exposes
  none of the events the template cares about (no kafka producer); its
  HTML responses are not actionable in that flow.
- **Practical rollback path.** Anyone needing to surface productpage's
  REST endpoint in Backstage can either (a) extend the specgen walker
  to discover an alternative `endpoints.go` location and map
  `text/html` operations to OpenAPI 3.1's `responses.<status>.content
  ['text/html']` form, or (b) refactor productpage to the canonical
  hex-arch layout and migrate it like the others. Neither is in scope
  for the initial rollout.

`productpage` therefore has no `services/productpage/api/` directory
and no `deploy/productpage/values-generated.yaml`. The Makefile's dual-
`-f` Helm install (`make k8s-deploy`) already short-circuits on missing
`values-generated.yaml`, so productpage's install line falls back to
the single `-f deploy/productpage/values-local.yaml` form transparently.
The CI `specs-drift` job naturally ignores productpage because the
runner's `DiscoverServices` only enumerates services that have one of
the declarative slices.
