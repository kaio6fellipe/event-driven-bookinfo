# DLQueue Service Design

## Overview

Dead Letter Queue service for the event-driven-bookinfo monorepo. Captures events that fail delivery after sensor retry exhaustion, persists them in PostgreSQL, and provides an API for inspection, replay, and resolution.

DLQueue is a CQRS service following the same hex arch patterns as all other services. It receives failed events through the Argo Events pipeline (sensor `dlqTrigger` → dlqueue EventSource webhook → Kafka EventBus → dlqueue sensor → dlqueue-write) and exposes a REST API for operators and downstream services.

## Problem

Currently, all three sensors (ratings, reviews, details) have a retry policy of 3 steps with 2s fixed duration. No `dlqTrigger` is configured. Events that fail after retries are silently lost. This is a critical production reliability gap.

## Architecture

### Event Flow

```
[Existing Sensor] ──(trigger fails after retries)──> [dlqTrigger]
    │
    ▼
[dlqueue EventSource webhook :12004] ──> [Kafka EventBus] ──> [dlqueue Sensor]
    │
    ▼
[dlqueue-write service] ──> [PostgreSQL]
    │
    ▼
[Operator/Service queries REST API on dlqueue-read]
    │
    ▼
[Replay: POST original_payload to original EventSource URL]
    │
    ▼
[Event re-enters full CQRS pipeline: EventSource → Kafka → Sensor → write service]
```

### Replay Model

Push model: dlqueue POSTs the `original_payload` with `original_headers` (including `traceparent`) to the stored `eventsource_url`. The event re-enters the full pipeline. All services must be idempotent since replayed events pass through the complete event pipeline, including notification triggers.

### Resolution Model

Explicit confirmation only. After replay, the DLQ record transitions to `replayed` with a `last_replayed_at` timestamp. Services or operators mark events as `resolved` via API. No magic timeouts.

If a replayed event fails again, the sensor's `dlqTrigger` fires again. DLQueue deduplicates on re-ingestion by matching on original event metadata (`sensor_name` + `failed_trigger` + `original_payload` hash), increments `retry_count`, and resets status to `pending`. Note: `event_id` (CloudEvents ID) cannot be used for dedup because Argo Events generates a new ID on each EventSource pass, including replays.

| `last_replayed_at` | timestamptz | Set on replay |

| `updated_at` | timestamptz | Last status change |

### State Machine

```
pending → replayed       (replay fired, sets last_replayed_at)
replayed → resolved      (service/operator confirms via API)
replayed → pending       (event returns from dlqTrigger, retry_count++)
pending → poisoned       (retry_count >= max_retries)
poisoned → pending       (operator resets, clears retry_count)
```

State transition validation is enforced on the domain entity via methods: `Replay()`, `Resolve(by)`, `ReIngest()`, `ResetPoison()`. Invalid transitions return `ErrInvalidTransition`.

### Domain Errors

- `ErrNotFound` — DLQ event not found
- `ErrAlreadyResolved` — event already resolved
- `ErrInvalidTransition` — invalid status transition

### Deduplication on Re-Ingestion

When a replayed event fails again and arrives back in dlqueue, the service matches on `sensor_name` + `failed_trigger` + `hash(original_payload)` (composite natural key). The CloudEvents `event_id` cannot be used because Argo Events generates a new ID on each EventSource pass. The payload hash ensures the same business event is identified regardless of transport envelope changes. If a match is found:
- Increment `retry_count`
- Reset `status` to `pending`
- Update `updated_at`
- If `retry_count >= max_retries` → set `status` to `poisoned`

If no match → create new record.

## Adapters

### Inbound: HTTP Handler

Standard handler pattern: parse input → validate → call service via inbound port → convert to DTO → write JSON response.

### Outbound: PostgreSQL Repository

- `pgxpool` connection pool, parameterized queries
- `original_payload` stored as JSONB
- `original_headers` stored as JSONB
- Indexes: `sensor_name + failed_trigger + payload_hash` (composite, for dedup), `status` (filtering), `event_source` (filtering), `created_at` (date range queries)
- Follows existing postgres adapter patterns (context-aware, domain error wrapping)

### Outbound: Memory Repository

In-memory implementation for tests and `STORAGE_BACKEND=memory`.

### Outbound: HTTP Replay Client

- Implements `EventReplayClient`
- POSTs `original_payload` to `eventsource_url`
- Forwards `original_headers` (preserves `traceparent` for trace correlation)
- Returns error on non-2xx response

### DLQueue Sensor

```yaml
spec:
  dependencies:
    - name: dlq-event-received-dep
      eventSourceName: dlq-event-received
      eventBusName: kafka
  triggers:
    - template:
        name: ingest-dlq-event
        http:
          url: http://dlqueue-write.bookinfo.svc.cluster.local/v1/events
          method: POST
          payload:
            - src: { dependencyName: dlq-event-received-dep, dataKey: body }
              dest: ""
      atLeastOnce: true
      retryStrategy:
        steps: 5
        duration: 2s
        factor: 2.0
        jitter: 1
```

Higher retry (5 steps + exponential backoff) — DLQ is the last safety net. No `dlqTrigger` on the DLQ sensor (recursive DLQ not supported).

### Existing Sensor Modifications

All 8 triggers across 3 sensors (ratings: 2, reviews: 4, details: 2) get:

1. `atLeastOnce: true` — required for dlqTrigger
2. Exponential backoff added to retry strategy (`factor: 2.0`, `jitter: 1`)
3. `dlqTrigger` block with HTTP trigger pointing to dlqueue EventSource

## System-Wide Prerequisite: Idempotency

DLQ replay pushes events through the full CQRS pipeline. All write services must be idempotent to handle duplicate delivery safely.

### Strategy: Idempotency Key + Natural Key Fallback

Each write service (reviews, ratings, details, notification, dlqueue) maintains a `processed_events` table:

```sql
CREATE TABLE processed_events (
    idempotency_key TEXT PRIMARY KEY,
    processed_at    TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

### Handler Flow

1. If request body contains `idempotency_key` → use it
2. If not → derive natural key from business payload:
   - reviews: `hash(product_id + reviewer + text)`
   - ratings: `hash(product_id + reviewer + stars)`
   - details: `hash(title + author + year + isbn)`
   - notification: `hash(recipient + subject + body + channel)`
   - dlqueue: `hash(sensor_name + failed_trigger + original_payload)` (from event metadata)
3. Check `processed_events` — if exists → return success (idempotent no-op)
4. Process + insert `idempotency_key` in single transaction

### Why CloudEvents `event_id` Cannot Be Used

Per the CNCF CloudEvents spec, intermediaries may regenerate the `id` field. Argo Events generates a new UUID (`uuid.New()`) for every POST to an EventSource webhook. On DLQ replay, the same payload gets a different CloudEvents `id`. The idempotency key must come from the business payload, not the transport envelope.

### Impact on Existing Services

- Request DTOs get `IdempotencyKey string \`json:"idempotency_key,omitempty"\`` field
- Service layer checks `processed_events` before processing
- Postgres adapter gets `processed_events` table (migration)
- Memory adapter tracks `map[string]bool` for local dev
- Productpage generates `idempotency_key` on form submission (optimization, not required)
- Each service is independently idempotent — no dependency on upstream
